Welcome to the Unified Client Framework and thank you for your contributions. This document will outline the process involved in making changes to the Unified Client Framework (UCF), previously known as the Spark Client Framework (SCF).

Getting Started

Please consult the following articles as needed to get started.

  1. Set up git to clone the repository and setting up your fork
  2. Building the Windows client
  3. Building the iOS client
  4. Building the Android client
  5. Building the macOS client
  6. Building the Linux client
  7. UCF Quick Start
  8. C++ Coding Guidelines, Qt Coding Guidelines and Qt Contributing Guide
  9. Architecture layers in UCF
  10. Official Color Palette

This unified client framework repository has three main parts, each of which can further be broken down as well, but at a high level, we have:

  1. spark-client-framework/ contains the C++ code for the services, adapters, models, data warehouse, networking and third-party code.
  2. common-head/ contains the view models that implement a specific user experience on top of the spark-client-framework services.
  3. products/ contains the individual products built on top of common-head, often implemented in the language of choice for that platform. For example, teams-ios is written primarily in Swift and is the iOS implementation of the Teams client. Similarly, teams-android is primarily written in Kotlin and is the Android implementation of the Teams client.

Contributing

First off, thank you for considering contributing to UCF! The contribution model of UCF is designed to facilitate efficient, safe, predictable and consistent contributions from the many different teams working on our collaboration products. Because of the varied time zones and geographies, care has been taken in designing this contribution model such that teams can mostly act autonomously, but, for the sake of all teams as a whole, equal care has also been taken to protect the investment in the shared code base.

We value:

  • Understanding and following our Code of Conduct. We expect all community members to treat others with respect and dignity.
  • Providing valuable feedback. We expect community members to participate in helping others improve the codebase.
  • Promoting a culture of learning. Be curious, ask questions and share what you learn.
  • Leading by example. We expect all community members to follow the same rules.
  • Upfront collaboration with design owners. Don’t wait until you open a PR to discuss a substantial change with a design owner.
  • Designing code to be testable and writing reliable tests. Testable code starts at design. We have a low tolerance for flaky tests.
  • Designs and code that are self-evident. Designs and code should be such that the patterns and intent is clear from reading the code.
  • Consistently formatted code (per language). No one likes reading code with a mix of styles.

Anyone with access to this repository is encouraged to make contributions. Read through the sections below and you are well on your way to making excellent contributions!

Engage Design Owners Early

If you are making changes to APIs, designs, algorithms or any other significant changes to the codebase, it is best to engage design owners (https://wiki.cisco.com/display/WX2/Design+owners) early. Doing so will almost always save you and others time. It can be demoralizing to a contributor to work hard on a contribution only to have it rejected during code review because some aspect of the design was not accounted for. It is also not a positive experience for a design owner to reject a contribution. It’s a win/win for you to consult design owners early.

Design owners are not considered dictators, but rather a recognized and respected expert in a specific area of the codebase. As engineers, we often have different ideas of the correct design. Please take the time to understand any concerns and advice a design owner may present to you before challenging their opinion. Once you’re confident that you understand their concerns, you can make a more informed opinion on why a different approach may be needed. Please consult the Code of Conduct as you progress through discussions (design owners and contributors alike).

Design owners are fluid, but documented below. Design owners are different from code owners, which are discussed later.

Write Good Code

All code is reviewed and approved before being merged into UCF and it can be frustrating as a reviewer to need ask for changes to a contribution when a contribution simply isn’t ready for review. It can also be frustrating to get review comments like these, so please be mindful of the following areas:

  1. Is your code style consistent with the rest of the code in that source file?
  2. Have you added tests as appropriate? (see Testing strategies)
  3. Is your contribution an appropriate size? (Big enough to have context, small enough to review effectively)
  4. Have you updated comments to ensure they reflect the state of the code?
  5. Have you removed dead code?

Write Good Commits

Good commit messages are an important aspect of maintaining a large software project. Please do your part in using commit messages to document your contribution. A good commit message will:

  1. Clearly summarize the contents of a commit on the first line in less than 72 characters.
  2. Use subsequent paragraphs to explain “what” and “why” vs “how”.
  3. Reference JIRAs as appropriate.

Separating your commits into atomic units of change is also important. You may prefer to make many small commits as you develop your code as a way of checkpointing, but these small commits are meaningless by themselves outsides of your development. Rebase your commits such that there is one commit per bug fix or small feature (large features will require multiple commits to keep them reviewable).

Create Good Pull Requests

If you’ve written a good commit message, you already have a good start on creating a good pull request (PR)! In fact, sometimes a good commit message is all you need for a good PR. However, always take the time to review the PR template and checklist to ensure you haven’t missed anything and add details as appropriate such that the reviewer will have all the necessary context to perform an informed review of the changes.

For PRs that change the UX of a product, include screenshots of the changes to give the reviewer context and a visual aid to the changes. Additionally, if possible and applicable, link to or otherwise reference a UX specification.

Once your PR is created, monitor the status checks that get kicked off in Jenkins for your PR. Status checks typically compile and test your code (merged with master) against all the platforms. The status check results will be added to your PR and will indicate if they passed, had test failures or failed completely. If you have a failing status check or failing tests, take some time to triage the failure before asking in the UCF Pipelines Build Squad space.

Typically, reviewers will not look at a PR until status checks are passing, so it’s in the best interest of your PR to address status check failures quickly.

Collaborate on your Pull Request

It’s time to have a conversation about your code! If you’ve been writing code professionally for years, this is nothing new, but if you’re a new graduate, this can feel intimidating at first. Please know that a reviewer’s intention is to learn about the code changes and to make suggestions to make it better. It’s within this process that we often grow the most, so lean into it and critically think about the suggestions being made. Keep in mind that reviewing code can also be uncomfortable as we’re being asked to provide suggestions, criticism and nitpicks on others code. If a reviewer is making nitpick type suggestions (e.g. code style, naming, etc.), please be mindful of these suggestions in subsequent contributions so nitpicks can largely be avoided.

The dance between a contributor and a reviewer can be a delicate one. Both sides should respect the time and effort that goes into creating a good contribution and performing a good review. Additionally, both sides should respect the motivations involved. A reviewer wants to ensure that each contribution is a quality contribution that leaves our shared codebase in a better state than it was before the contribution. A contributor is likewise motivated, but might also be getting pressure to deliver a bug fix or feature on a deadline. While a tight schedule is never an excuse to merge bad code or bad design patterns, the reviewer can take this into consideration while requesting changes to happen immediately within the PR, immediately following the PR, or at some later agreed date.

Merging a Pull Request

Pull requests can be merged once all status checks are passing and you have at least one review approval. The approval must be from someone that has attained committer status and, depending on the code changed, may need to be from specific groups of code owners.

If you have attained committer status, you may merge your own code after the PR has been approved by another committer and the status checks pass. Otherwise, a committer will merge the PR for you. If they don’t merge it for you but have approved the PR, ping the approver to have them merge the PR.

Becoming a Committer

If you contribute to UCF regularly, you might be asking yourself how you attain committer status and that’s a great question! We want all regular contributors to become committers but we also want to ensure that each contributor has gained enough experience contributing to understand the designs in place and the values we embody. With every great privilege, comes great responsibility and earning committer status on the UCF repository is no different. As a committer, it is your responsibility, through code reviews and collaboration with others, to ensure that others uphold these standards and values.

Becoming a committer requires that you consistently contribute significant quality changes to the UCF codebase. To avoid having a committee of ivory tower committers deciding who “consistently contributes significant quality changes,” this process is automated. While it is possible to game the system and gain committer access earlier than you otherwise would, it is very highly encouraged to contribute to UCF at your own pace and use this time to truly learn the code, design and culture of the UCF codebase and community.

The criteria used to promote contributors to committers is this simple algorithm. Every day, all PRs from the last six months are reviewed and we count the number of PRs each contributor has opened that 1) were merged 2) were significant by counting the lines changed and 3) had no reviews that explicitly requested changes. If that number of PRs is >= 6, then a contributor is promoted to committer. Many contributors, depending on their cadence of contributions, will be promoted to committer within a few weeks of their first contribution to UCF. If that includes you, congratulations and welcome!

Design Owners and Code Owners

While the UCF community itself is the owner of the codebase, we do promote some notion of ownership within the codebase. As previously mentioned, this ownership is not meant to be some sort of dictatorship. Instead, consider these owners to be our experts that should be consulted regularly and their input given very special consideration.

A design owner can save you tremendous time by suggesting a specific change or implementation to a given problem. Engaging with them early is the best way to ensure that you are about to embark on the right track. It can be very frustrating to discover that the code you just spent a week writing cannot be merged or accepted because it does not fit in with the designs in place or because of other fundamental concerns.

When a design owner suggests an implementation or design that you think isn’t workable or if you think a different approach would be better, engage with them on the technical merits in a positive exchange of ideas. If there’s a difference of opinions that cannot be resolved in a timely manner, engage the larger group of design owners that are coordinated by Kevin Gallagher (@kevigall).

Code owners are different than a design owner. Code owners are a group of committers that are experts in a specific area of the codebase. While that sounds very similar to a design owner, it’s in how code owners are consulted that is the difference. A group of code owners is automatically assigned to review your PR based on the files you change. This ensures that your PR is reviewed by an engineer with the right experience and expertise. For example, if you open a PR that modifies any android specific files, your PR will require an approval from a committer that is also regarded as having android specific experience. Code owners are codified in GitHub in the CODEOWNERS file in the root of the repository.
